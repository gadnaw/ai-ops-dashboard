# Plan 01-01: Scaffold — Next.js 15 + Prisma + Supabase + Folder Structure

## Objective

Bootstrap the project's entire technical substrate from scratch. This plan produces a runnable Next.js 15 application with TypeScript strict mode, Tailwind CSS 4, Prisma 7 with dual connection strings pointing at Supabase, the canonical folder structure all future phases write into, and the initial database schema for `profiles` and `api_keys` tables.

No UI features are built here. The output is infrastructure: a dev server that starts cleanly, a Prisma schema that migrates without error, and documented conventions that every subsequent plan follows.

**`docs/ADR-001-architecture-decisions.md`** — document all architectural decisions made in this phase:

```markdown
# ADR-001: Architecture Decisions — Foundation

## Status: Accepted

## Context

Establishes conventions all subsequent phases must follow. Deviating from these decisions requires a new ADR.

## Decisions

### Folder Structure

```
src/
  app/                    # Next.js App Router routes
    (auth)/               # Auth pages — login, signup (no shared layout chrome)
    (dashboard)/          # Protected dashboard pages (shared nav/sidebar)
    api/                  # API routes
      v1/                 # Public API (external consumers)
  lib/                    # Server-side services and utilities
    db/                   # Database layer
      prisma.ts           # Singleton Prisma client (canonical import)
    auth/                 # Auth helpers — session, role guards
    model-router/         # Multi-model routing engine (Phase 2)
      registry.ts         # Provider registry (canonical import)
  components/             # Shared React components
    ui/                   # Primitive UI components (buttons, inputs, cards)
prisma/
  schema.prisma           # Single schema file
  migrations/             # Prisma migrate output
  seed.ts                 # Seed script (Phase 2)
docs/
  ADR-001...              # Architecture decision records
```

### Naming Conventions

- Files: `kebab-case.ts` for all files
- React components: `PascalCase` function, `kebab-case` filename
- Database tables: `snake_case`
- TypeScript types/interfaces: `PascalCase`
- Environment variables: `SCREAMING_SNAKE_CASE`
- API routes: `/api/v1/resource` pattern

### Access Pattern Split (Pitfall 13 — RLS + Prisma)

- **Prisma** is used for ALL server-side data access (Server Components, Server Actions, API routes).
  Application-level auth checks are performed before any Prisma query.
- **Supabase JS client** is used ONLY for client-side real-time subscriptions (Supabase Realtime).
  Client component subscriptions use `supabase.channel()` — never for data fetching.
- RLS policies are a defense-in-depth layer, not the primary auth guard.

### Package Manager

`pnpm` only. Do not use `npm` or `yarn` in any script, README, or CI config.

### TypeScript Strictness

`strict: true` plus `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noImplicitReturns`.
`any` types are banned in production code. Use `unknown` + type guards.
```

**`prisma/schema.prisma`** — complete initial schema with dual connection strings, profiles, and api_keys:

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider  = "postgresql"
  // Pooled connection for runtime queries (PgBouncer, port 6543)
  // connection_limit=1 required for serverless to avoid connection exhaustion
  url       = env("DATABASE_URL")
  // Direct connection for migrations (port 5432, bypasses PgBouncer)
  directUrl = env("DIRECT_URL")
}

// ----- USER IDENTITY -----
// Mirrors Supabase auth.users. All application FKs reference profiles(id).
// DO NOT reference auth.users(id) directly from application tables.
model Profile {
  id        String   @id @db.Uuid
  email     String   @unique
  role      Role     @default(VIEWER)
  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz

  apiKeys   ApiKey[]

  @@map("profiles")
}

enum Role {
  ADMIN
  DEVELOPER
  VIEWER

  @@map("role")
}

// ----- API KEY MANAGEMENT -----
// Plaintext key shown once at creation. Only SHA-256 hash persisted.
// Per-key rate limit overrides: null means use global defaults.
// Phase 4 rate_limit_events table has FK to api_keys(id).
model ApiKey {
  id             String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  profileId      String    @map("profile_id") @db.Uuid
  label          String
  // SHA-256 hash of the plaintext key (hex string, 64 chars)
  keyHash        String    @unique @map("key_hash")
  // Display prefix only — e.g. "sk-...abc4" (first 4 + last 4 chars of plaintext)
  keyPrefix      String    @map("key_prefix")
  // Per-key rate limit overrides. NULL means use global defaults.
  rateLimitRpm   Int?      @map("rate_limit_rpm")
  rateLimitTpm   Int?      @map("rate_limit_tpm")
  expiresAt      DateTime? @map("expires_at") @db.Timestamptz
  lastUsedAt     DateTime? @map("last_used_at") @db.Timestamptz
  revokedAt      DateTime? @map("revoked_at") @db.Timestamptz
  createdAt      DateTime  @default(now()) @map("created_at") @db.Timestamptz
  updatedAt      DateTime  @updatedAt @map("updated_at") @db.Timestamptz

  profile        Profile   @relation(fields: [profileId], references: [id], onDelete: Cascade)

  @@index([profileId])
  @@index([keyHash])
  @@map("api_keys")
}
```

**Step 3 — Create the `profiles` table trigger in Supabase SQL Editor:**

The `profiles` table is managed by Prisma, but it needs a Supabase trigger to auto-create a profile row when a new auth.users record is inserted. Run this SQL in Supabase SQL Editor (Project > SQL Editor):

```sql
-- Create extension for UUID generation (already enabled by default in Supabase)
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Trigger function: auto-create profile when new auth user signs up
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  INSERT INTO public.profiles (id, email, role, created_at, updated_at)
  VALUES (
    NEW.id,
    NEW.email,
    'VIEWER',  -- Default role; Admin promotes manually
    NOW(),
    NOW()
  )
  ON CONFLICT (id) DO NOTHING;
  RETURN NEW;
END;
$$;

-- Attach trigger to auth.users
DROP TRIGGER IF EXISTS on_auth_user_created ON auth.users;
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

**Step 4 — Enable Row Level Security on profiles and api_keys:**

Run in Supabase SQL Editor after migration:

```sql
-- Enable RLS on profiles
ALTER TABLE public.profiles ENABLE ROW LEVEL SECURITY;

-- Policy: users can read their own profile
CREATE POLICY "profiles_select_own"
  ON public.profiles
  FOR SELECT
  USING (auth.uid() = id);

-- Policy: users can update their own profile (email, not role)
CREATE POLICY "profiles_update_own"
  ON public.profiles
  FOR UPDATE
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);

-- Policy: Admins can read all profiles (for user management)
CREATE POLICY "profiles_select_admin"
  ON public.profiles
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.profiles
      WHERE id = auth.uid() AND role = 'ADMIN'
    )
  );

-- Enable RLS on api_keys
ALTER TABLE public.api_keys ENABLE ROW LEVEL SECURITY;

-- Policy: users can read their own API keys
CREATE POLICY "api_keys_select_own"
  ON public.api_keys
  FOR SELECT
  USING (profile_id = auth.uid());

-- Policy: users can create their own API keys
CREATE POLICY "api_keys_insert_own"
  ON public.api_keys
  FOR INSERT
  WITH CHECK (profile_id = auth.uid());

-- Policy: users can revoke their own API keys
CREATE POLICY "api_keys_update_own"
  ON public.api_keys
  FOR UPDATE
  USING (profile_id = auth.uid())
  WITH CHECK (profile_id = auth.uid());

-- Policy: Admins can manage all API keys
CREATE POLICY "api_keys_admin_all"
  ON public.api_keys
  FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM public.profiles
      WHERE id = auth.uid() AND role = 'ADMIN'
    )
  );
```

## provides_interface Detail

### PrismaClient
- **Import:** `import { prisma } from '@/lib/db/prisma'`
- **Used by:** All server-side code in all phases (Server Components, Server Actions, API routes)
- **Pattern:** Singleton — one PrismaClient per Node.js process, reused across hot reloads in dev
- **Do not:** Call `new PrismaClient()` outside of this file. Do not use the Supabase JS client for DB queries.

### DatabaseSchema
- **Tables created:** `profiles`, `api_keys`
- **Extension pattern:** Each subsequent phase adds tables via new `prisma migrate dev` runs. Never modify existing columns without a migration.
- **FK convention:** All user identity FKs reference `profiles(id)`, not `auth.users(id)` directly.
- **api_keys FK:** Phase 4 adds `rate_limit_events` with `api_key_id UUID REFERENCES api_keys(id)`. The `api_keys.id` column is UUID — ensure Phase 4 FK is typed as UUID.
- **RLS note:** RLS is enabled via SQL (Task 3). Prisma bypasses RLS when using the service role key. Application-level auth checks in Server Actions and API routes are the primary guard.

### FolderConventions
- **ADR location:** `docs/ADR-001-architecture-decisions.md`
- **Key rule:** `src/lib/` is server-only. Never import from `src/lib/` in Client Components.
- **Prisma import path:** `@/lib/db/prisma` — non-negotiable. Phase 4+ code that uses `@/lib/prisma` is wrong.
- **Model router import path:** `@/lib/model-router/registry` — established in Phase 2, referenced here for documentation.

### EnvironmentConfig
- **Validation:** `src/lib/env.ts` using @t3-oss/env-nextjs. Import `env` object instead of raw `process.env` in application code.
- **New env vars in later phases:** Add to `src/lib/env.ts` server/client schema and to `.env.example` with descriptions.
- **Dual connection strings:** `DATABASE_URL` always has `?pgbouncer=true&connection_limit=1`. `DIRECT_URL` never has PgBouncer parameters.

### PackageScripts
- `pnpm db:migrate` — CI/CD migration (deploy mode, no prompts)
- `pnpm db:migrate:dev` — local development migration (creates migration files)
- `pnpm db:generate` — regenerate Prisma client after schema changes
- `pnpm db:studio` — open Prisma Studio for DB inspection
- `pnpm test` — run Vitest in watch mode
- `pnpm test:run` — run Vitest once (CI)
- `pnpm test:e2e` — run Playwright E2E suite
